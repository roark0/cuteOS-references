Abstract

This is a collection of technical notes on the Universal Serial Bus (USB).

For a more general introduction, see the Wikipedia entry for Universal Serial Bus.

Electrical Interface

Bus States

General Summary
Bus State D+ D-
Differential "1" high low
Differential "0" low high
Single-Ended 0 (SE0) low low
Single-Ended 1 (SE1) high high


Differential "1"

For low-/full-speed devices, a Differential "1" is asserted as follows: 

 D+ > VOH (min) and D- < VOL (max)

where VOH (min) = 2.8 V and VOL (max) = 0.3 V

When the bus is not actively driven and a full-/high-speed device is attached, a pull-up resistor on D+ puts the bus in the Differential "1" state.

Differential "0"

For low-/full-speed devices, a Differential "0" is asserted as follows: 

 D- > VOH (min) and D+ < VOL (max)

where VOH (min) = 2.8 V and VOL (max) = 0.3 V

When the bus is not actively driven and a low-speed device is attached, a pull-up resistor on D- puts the bus in the Differential "0" state.

Single-Ended 0 (SE0)

For low-/full-speed devices, a Single-Ended 0 (SE0) is asserted as follows: 

 D+ and D- < VOL (max)

where VOL (max) = 0.3 V

When the bus is not actively driven and no device is attached, pull-down resistors on D-/D+ put the bus in the SE0 state.

Single-Ended 1 (SE1)

For low-/full-speed devices, a Single-Ended 1 (SE1) is asserted as follows: 

 D+ and D- > VOSE1 (min)

where VOSE1 (min) = 0.8 V

Data J

For a low-speed device, the Data J state is the differential "0".

For a full-speed device, the Data J state is the differential "1".

Data K

For a low-speed device, the Data K state is the differential "1".

For a full-speed device, the Data K state is the differential "0".

Idle

When the bus is not actively driven, pull-down resistors within the host and a pull-up resistor within the device maintain the Data J state.

If the bus is idle for more than 3 ms, devices are requird to enter a low-power "suspended" state.

Resume

Reset

The Reset state is defined as the SE0 state persisting for at least 10 ms.

Device States

Attached

The device is attached to the bus but VBUS is not active (i.e., it is not powered).

Powered

VBUS is active but the device has not been reset.

Default

The device has been reset but it has not been assigned an address.

Address

The device has been assigned an address but a configuration has not been selected.

Configured

The device has been configured and is ready for operation.

Suspended

The device enters this state whenever the bus has been idle for at least 3 ms.

Requests

Common Request Format

Common Request Format Summary
Offset Field Size Description
0 bmRequestType 1 Request type flags
1 bRequest 1 Constant identifying which request this is
2 wValue 2 Meaning depends on the value of bRequest
4 wIndex 2 Meaning depends on the value of bRequest
6 wLength 2 Number of bytes to transfer in the data stage

bmRequestType

Flags indicating the request's type, and recipient, and data transfer direction:

Bit 7:
Direction
0 = Host-to-device
1 = Device-to-host
Bits 6...5:
Type
00b = Standard
01b = Class
10b = Vendor
Bits 4...0:
Recipient
00000b = Device
00001b = Interface
00010b = Endpoint
00011b = Other

Bit seven indicates the direction in which data is transferred in the data stage. This field is ignored if wLength is zero (indicating there is no data stage).

bRequest

This field identifies which specific request this is. Its meaning depends on the "Type" bits (6 and 5) in bmRequestType. If these bits are set to "Vendor" (10b), the bRequest field is defined by the device manufacturer. If these bits are set to "Class" (01b), this field is defined by a class specification. If these bits are set to "Standard" (00b) this field is defined as follows:

0x00: GET_STATUS
0x01: CLEAR_FEATURE
0x03: SET_FEATURE
0x05: SET_ADDRESS
0x06: GET_DESCRIPTOR
0x07: SET_DESCRIPTOR
0x08: GET_CONFIGURATION
0x09: SET_CONFIGURATION
0x0A: GET_INTERFACE
0x0B: SET_INTERFACE
0x0C: SYNCH_FRAME

wValue

The meaning of this field depends on the request.

wIndex

The meaning of this field depends on the request. Its value usually represents an index or an offset.

wLength

The number of bytes to transfer during the data stage, or zero if there is no data stage.

If the "Direction" bit (7) of bmRequestType is set (indicating a device-to-host request), the device may send up to wLength bytes. If the direction bit is reset (indicating a host-to-device request), the host will send exactly wLength bytes.

Get Configuration

USB 2.0 Section 9.4.2

"Get Configuration" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 10000000b GET_CONFIGURATION
(0x08) 		0x0000 0x0000 0x0001 Configuration
Value

Get the current device configuration identifier as set by Set Configuration, or zero if not configured.

The returned value corresponds to the bConfigurationValue field of the configuration descriptor to which the device is currently set.

Get Descriptor

USB 2.0 Section 9.4.3

"Get Descriptor" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 10000000b GET_DESCRIPTOR
(0x06) 		MSB = Descriptor Type
LSB = Descriptor Index 		Zero or
Language ID 		Descriptor
Length 		Descriptor

This request retrieves the specified descriptor.

For a standard "Get Descriptor" request, the high byte of wValue may be one of three values:

DEVICE (0x01)

Get_Descriptor(DEVICE) returns the one and only device descriptor.

The low byte of wValue is always zero.

wIndex is always zero.

CONFIGURATION (0x02)

Get_Descriptor(CONFIGURATION) returns the specified configuration descriptor as well as all interface descriptors and endpoint descriptors for that configuration. These descriptors are returned in a hierarchical order. For example, if the configuration has two interfaces, where the first interface has one endpoint and the second interface has two, the descriptors are returned in the following order:

  • Configuration descriptor
    • Interface descriptor
      • Endpoint descriptor
    • Interface descriptor
      • Endpoint descriptor
      • Endpoint descriptor

The low byte of wValue identifies the configuration whose descriptors are to be retrieved. This must be between one and the value of the bNumConfigurations field of the device descriptor (inclusive).

wIndex is always zero.

If the host only wants the configuration descriptor, it should set the wLength field to 10 (the size of a standard configuration descriptor) and the device will not return the interface or endpoint descriptors.

STRING (0x03)

Get_Descriptor(STRING) returns the specified string descriptor.

The low byte of wValue contains the index of the string descriptor to be retrieved.

wIndex contains a language identifier (zero if retrieving string descriptor zero).

High-speed devices additionally support the DEVICE_QUALIFIER (0x06) and OTHER_SPEED_CONFIGURATION (0x07) descriptors.

The host sets the wLength field to the maximum number of bytes to return.

If an unsupported descriptor is requested, a request error is generated.

Get Status

USB 2.0 Section 9.4.5

"Get Status" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 10000000b GET_STATUS
(0x00) 		0x0000 0x0000 0x0002 Status
Interface 10000001b Interface number
Endpoint 10000010b Endpoint address

The definition of the Status field depends on the whether the recipient is a device, interface, or endpoint.


Recipient = Device (bmRequestType = 10000000b):

D15...2 Reserved (always zero)
D1 Remote Wakeup
D0 Self Powered

D1 is set to one if the device is currently enabled to perform remote wakeup. D0 is set to one if the device is currently self powered (D0 is reset to zero if the device is currently bus powered).


Recipient = Interface (bmRequestType = 10000001b):

D15...0 Reserved (always zero)

No status information is reported for an interface.


Recipient = Endpoint (bmRequestType = 10000010b):

D15...1 Reserved (always zero)
D0 Halt
D0 is set to one if the endpoint is currently halted.

Set Address

USB 2.0 Section 9.4.6

"Set Address" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 00000000b SET_ADDRESS
(0x05) 		Device
Address 		0x0000 0x0000 N/A

This request sets the device's address.

When a device is reset, it responds at the default address (0x00). The host then issues a Set Address request to assign a new (non-zero) address. The device thus transitions from the "Default" state to the "Address" state and is ready to be configured.

Set Configuration

USB 2.0 Section 9.4.7

"Set Configuration" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 00000000b SET_CONFIGURATION
(0x09) 		Configuration
Value 		0x0000 0x0000 N/A

This request sets the device's configuration. The "Configuration Value" corresponds to the value of the bConfigurationValue field of a configuration descriptor.

After the host assigns an address to the device (see Set_Address), its next task is to select a configuration. The host determines which configurations the device supports by reading its configuration descriptors. It can then enable the desired configuration using the Set_Configuration command. The device thus transitions from the "Address" state to the "Configured" state.

Set Interface

USB 2.0 Section 9.4.10

"Set Interface" Request Format
bmRequestType bRequest wValue wIndex wLength Data
Device 00000001b SET_INTERFACE
(0x0B) 		Alternate Setting Interface Number 0x0000 N/A

This request selects the settings to be used for the specified interface within the current configuration.

wValue and wIndex are set to the values of bAlternateSetting and bInterfaceNumber fields, respectively, of a previously returned interface descriptor.

If the interface only supports one group of settings, the status stage of this request may return STALL. If wValue or wIndex are not valid, a request error is generated.

By default, alternate setting 0 is assumed for each interface.

bLength

The total length of the descriptor in bytes, including this field. For a device descriptor, this is always 18 bytes.

bDescriptorType

This field is set to the DEVICE constant (0x01) indicating this is a device descriptor.

bcdUSB

Binary coded decimal (BCD) representation of the USB specification version to which this device conforms. This is encoded as 0xJJMN for version JJ.M.N. EG, USB 2.0 devices should set this field to 0x0200.

Microsoft Windows 2000 and later accepts the following as valid values for this field:

  • 0x0100 (USB v1.0) (not valid if the device is a hub)
  • 0x0110 (USB v1.1)
  • 0x0200 (USB v2.0)

USB FAQ: Introductory Level

bDeviceClass

This represents the class of the device as a whole. If this device does not belong to a class or each interface operates independantly (i.e., a composite device), this field is reset to zero.

Class codes are assigned by USB-IF and are in the range 0x01 to 0x0FE (inclusive). 0xFF is the code reserved for vendor-defined classes. Click here for the complete list.

This field is rarely used except by communication devices. Most functional classes are defined at the interface level (see bInterfaceClass).

This field helps the host determine which driver(s) should be loaded for this device.

bDeviceSubClass

This represents the subclass of the device as a whole, whose meaning depends on the value of bDeviceClass. If bDeviceClass is reset to zero, bDeviceSubClass must be reset to zero. If the class code is 0xFF (vendor-defined), the subclass code is also vendor-defined. For all other class codes (those in the range 0x01 to 0xFE, inclusive), the valid subclass codes are maintained here by USB-IF.

This field helps the host determine which driver should be loaded for this device.

bDeviceProtocol

A protocol code assigned by USB-IF whose meaning depends on the values of bDeviceClass and bDeviceSubClass. The protocol code 0xFF indicates a vendor-defined protocol.

This field helps the host determine which driver should be loaded for this device.

bMaxPacketSize0

Maximum packet size for endpoint zero. This must be 8, 16, 32, or 64 for low- and full-speed devices, or 64 for high-speed devices.

idVendor

A unique identifier of the device's manufacturer. Vendor IDs are assigned by USB-IF at a cost of $1500/each for non-members. Some USB chip manufacturers will let you use their vendor ID with their chips, and assign you a block of product IDs.

This field helps the host determine which driver should be loaded for this device.

idProduct

A unique product identifier. Product IDs are assigned by the owner of the vendor ID being used.

This field helps the host determine which driver should be loaded for this device.

bcdDevice

Device release version in binary-coded decimal (BCD).

Though the exact format of this field isn't specified in the standard, the common use seems to be 0xMMNN representing version MM.NN. EG, version 3.51 would be encoded as 0x0351.

This field helps the host determine which driver should be loaded for this device.

iManufacturer

Index of a string descriptor describing the manufacturer. This is optional and should be zero if not used.

iSerialNumber

Index of a string descriptor containing the device's serial number. This is optional and should be zero if not used.

bNumConfigurations

Number of supported configurations.

The host uses this field to determine how many configuration descriptors to retrieve.

USB 2.0 Section 9.6.3

Configuration Descriptor Summary
Offset Field Size Description
0 bLength 1 Size of this descriptor in bytes (9)
1 bDescriptorType 1 CONFIGURATION constant (0x02)
2 wTotalLength 2 Total size in bytes of all descriptors returned for this configuration
4 bNumInterfaces 1 Number of interfaces used in this configuration
5 bConfigurationValue 1 A one-based index identifying this configuration

7 bmAttributes 1 Attribute flags

bLength

The total length of the descriptor in bytes, including this field. For a configuration descriptor, this is always 9 bytes.

bDescriptorType

This field is set to the CONFIGURATION constant (0x02) indicating this is a configuration descriptor.

wTotalLength

The total length of all descriptors returned in a Get_Descriptor(CONFIGURATION) request. This includes this configuration descriptor itself plus all its interface and endpoint descriptors. This may also include class- and vendor-specific descriptors.

bNumInterfaces

Number of interfaces used in this configuration.

The value of this field should match the number of interface descriptors returned in a Get_Descriptor(CONFIGURATION) request.

bConfigurationValue

A non-zero unique identifier for this configuration.

Get_Configuration and Set_Configuration requests use this value to specify a configuration.

iConfiguration

Index of a string descriptor describing this configuration. This is optional and should be zero if not used.

bmAttributes

A bitmap defined as follows:

D7 Reserved (always one)
D6 Self-powered
D5 Remote Wakeup
D4...0 Reserved (always zero)

D0-D4 are always zero and D7 is always one. D6 is set to one if the device is capable of powering itself in this configuration. D5 is set to one if the device supports remote wakeup in this configuration.

bMaxPower

Specifies the maximum amount of current the device draws from the bus in this configuration, expressed in units of 2 mA. EG, a 100 mA device would specify a bMaxPower value of 50.

Endpoint Descriptor

USB 2.0 Section 9.6.6

Endpoint Descriptor Summary
Offset Field Size Description
0 bLength 1 Size of this descriptor in bytes (7)
1 bDescriptorType 1 ENDPOINT constant (0x05)
2 bEndpointAddress 1 This endpoint's address
3 bmAttributes 1 Endpoint attribute flags
4 wMaxPacketSize 2 Maximum packet size and related flags
6 bInterval 1 Polling interval

bLength

The total length of the descriptor in bytes, including this field. For an endpoint descriptor, this is always 7 bytes.

bDescriptorType

This field is set to the ENDPOINT constant (0x05) indicating this is an endpoint descriptor.

bEndpointAddress

This endpoint's address, which consists of an endpoint number and direction:

Bit 3...0: Endpoint number
Bit 6...4: Reserved (always zero)
Bit 7: Direction

Bit 7 is set to one for an IN endpoint, or reset to zero for an OUT endpoint. This bit is ignored for control endpoints.

Endpoint zero is the default control endpoint and does not have a descriptor. Therefore, the endpoint number portion of this field will never be zero.

bmAttributes

Endpoint attribute flags, defined as follows:

Bits 1...0: Transfer Type
00 = Control
01 = Isochronous
10 = Bulk
11 = Interrupt
Bits 3...2: Synchronization Type
00 = No synchronization
01 = Asynchronous
10 = Adaptive
11 = Synchronous
Bits 5...4: Usage Type
00 = Data endpoint
01 = Feedback endpoint
10 = Implicit feedback data endpoint
11 = Reserved
Bits 7...6: Reserved (always zero)

If transfer type is not isonchronous, synchronization type and usage type should be reset to zero by the peripheral and ignored by the host.

wMaxPacketSize

Maximum packet size supported by this endpoint, as well as related flags:

Bits 10...0: Maximum packet size (bytes)
Bits 12...11: Additional transaction opportunities per microframe
Bits 15...13: Reserved (always zero)

Bits 12...11 are only valid for high-speed isochronous and interrupt endpoints.

bInterval

Polling interval in frames or microframes.

String Descriptor

USB 2.0 Section 9.6.7

String Descriptor Summary
Offset Field Size Description
0 bLength 1 Size of this descriptor in bytes (variable)
1 bDescriptorType 1 STRING constant (0x03)
2 wLANGID or bString variable Language ID array or unicode string

bLength

The total length of the descriptor in bytes, including this field.

bDescriptorType

This field is set to the STRING constant (0x03) indicating this is a string descriptor.

wLANGID

String descriptor zero consist of an array of two-byte language code identifiers (LANGIDs) indicating which languages are supported by this device.

The host retrieves string descriptor zero to determine which languages the device supports, then specifies one of these LANGIDs when retrieving other string descriptors.

U.S. English is LANGID 0x0409. A complete list of LANGIDs is maintained in the "USB Language Identifiers" specification.

bString

String descriptors other than string descriptor zero consist of a (non NULL terminated) UNICODE string.

Features

USB devices have certain features that may be enabled/disabled by the host. Enabling a feature is accomplished by issuing a Set Feature request containing a feature selector constant indicating which feature is to be enabled. Similarly a feature is disabled using the Clear Feature request.

Three standard feature selectors are defined:

0x00 ENDPOINT_HALT
0x01 DEVICE_REMOTE_WAKEUP
0x02 TEST_MODE

ENDPOINT_HALT applies only to endpoints, and DEVICE_REMOTE_WAKEUP and TEST_MODE apply only to devices.

Endpoint Halt

When a communication error occurs, the endpoint becomes halted. In this state it responds to all transactions with a STALL handshake, indicating an error has occured. This condition must be explicitly cleared by the host using the Clear_Feature(ENDPOINT_HALT) request. The host is also allowed to put an endpoint in the halt state by issuing a Set_Feature(ENDPOINT_HALT) request.

The host may determine whether or not an endpoint is halted using the Get_Status request.

Device Remote Wakeup

When DEVICE_REMOTE_WAKEUP is enabled, the device is allowed to request a suspended host to resume operation. Neither devices nor hosts are required to support this feature.

A device that supports remote wakeup indicates this in the bmAttributes field of its configuration descriptor. By default, this feature is disabled and the device is not allowed to signal remote wakeup. If the host also supports this feature, it may enable it within the device by issuing a Set_Feature(DEVICE_REMOTE_WAKEUP) request.

The device performs remote wakeup by putting the bus in the resume state for 1-15 ms. This is the Data K state, which is defined as D+ > VOH (min) and D- < VOL (max) for low-speed devices, or D- > VOH (min) and D+ < VOL (max) for high-speed devices. The device is allowed to do this only when the DEVICE_REMOTE_WAKEUP feature is enabled, and the bus has been in the idle state for at least 5 ms.

Test Mode

Composite Devices

A USB composite device is one that has multiple independant interfaces. When a host enumerates a composite device, it loads a generic "multi-interface" driver for the device, which creates a PDO for each interface on the device. This way, drivers may be re-used at the interface-level. For example, a combination keyboard/mouse implemented as a composite device may be supported by stand-alone keyboard and mouse drivers instead of requiring an additional all-in-one keyboard+mouse driver.

The bDeviceClass and bDeviceSubClass fields in a composite device's device descriptor should be zero.

Certain versions of Windows 2000 hang if a composite device is removed while being reset (Microsoft Knowledge Base Article 278489);

Serial Numbers

A USB device may be programmed with a serial number, allowing the host to keep track of different instances of products with the same vendor ID, product ID, and device release number. The serial number is stored as a string descriptor, and the index of that string descriptor is stored in the iSerialNumber field of the device descriptor. If a serial number is not assigned to the device, iSerialNumber is zero.

Microsoft requires that if a serial number is set it must be unique, and each byte in the serial number must be in the range 0x20 <= N <= 0x7F, excluding 0x2C. Windows operating systems always use LANGID 0x0409 to retrieve serial numbers.
